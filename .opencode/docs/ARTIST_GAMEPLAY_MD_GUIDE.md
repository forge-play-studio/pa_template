# Artist Gameplay MD Guide

本文档定义美术如何配合 `gameplay` agent 生成和修正 `gameplay.md`。它只服务于文档整理，不要求 agent 实现 gameplay 代码。

## 1. 基本原则

美术不需要写程序方案，但需要把目标玩法流程、资产关系和地编意图说清楚。

参考 HTML / playable 是可选输入：

```text
有参考时：说明目标项目哪些流程来自参考、哪些需要替换、哪些允许调整。
无参考时：直接描述目标项目 gameplay 流程。
参考不清楚时：以美术人工观察和确认的目标流程为准。
```

AI 不应把自己对参考 HTML 的推断当成确定需求。文档中必须区分：

```text
需求方明确要求
美术确认
AI 推测
团队默认规则
待确认项
```

## 2. 美术应提供的目标流程

agent 应优先引导美术按玩家视角回答：

```text
开局玩家看到什么？
玩家第一步做什么？
玩家如何获得第一个资源？
资源放到哪里？
哪些资源会被玩家可视化背在角色身后？哪些资源不会背在身后？
资源如何变成钱、订单进度或关卡目标？
第一次升级或解锁是什么？
升级后场景或自动化发生什么变化？
是否出现 NPC、顾客、车辆、机器、工人或队列？
玩家完成到哪一步算 first playable 结束？
CTA / Endcard / 跳转什么时候出现？
```

## 3. 资产与 Gameplay Object 关系

美术应说明每个重要资产在 gameplay 中的角色：

```text
玩家主体
资源
资源来源
容器 / 堆放点
加工点
售卖 / 提交点
升级 / 解锁点
NPC / 顾客 / 工人
路径点 / 等待点
运行时生成点
Decoration
```

不能把缺少规则的 gameplay 对象临时写成 Decoration。Decoration 只能表示不参与 gameplay 的装饰对象。

如果某个资产会参与 first playable 主路径，美术还应说明它是否需要：

```text
gameplay binding
zone
spawn point
path point
runtime parent
角色身后背负挂点 / 堆叠表现
guide target
unlock / hide / show 状态
```

如果这些信息缺失，agent 必须追问用户；不能把 AI 对节点名或资产名的猜测写成已确认 binding。尤其不能从资源资产存在、背包存在或 inventory 规则中推断该资源可以背在角色身后。

agent 整理 `gameplay.md` 时，必须把美术资产和场景节点关系整理成可供 gameplay builder 使用的绑定索引，至少包括：

```text
可复用美术资产：玩法对象使用哪个 asset id、GLB、贴图、音效或资源文件。
玩法对象到场景节点：每个玩法对象对应哪个 scene node、zone 或配置 id。
区域 / 锚点 / 路径点：触发区、出生点、拾取区、支付区、输入输出锚点、等待点、离开点、引导目标。
堆放和排列：玩家身后背负、场景资源堆、机器输入输出、订单装载、支付现金的高度、排列、容量或无容量规则。
绑定还原要求：哪些必须按节点 ID 绑定，哪些可用 asset 动态生成，哪些只是视觉参考或 decoration，哪些 fallback 已被用户确认。
```

## 4. 地编意图

美术应补充场景布局背后的玩法意图：

```text
玩家出生点
第一段引导方向
资源区
交付 / 售卖 / 加工区
升级 / 解锁区
NPC / 顾客 / 车辆的出生、等待、工作和离开位置
初始隐藏或锁定的区域
解锁后新增、消失或改变的区域
```

这些信息不等于最终 binding 表，但应进入 `gameplay.md`，作为后续地编绑定和开发实现的依据。

## 5. 生成后的完整性检查

写完或更新 `gameplay.md` 后，agent 应检查：

```text
是否有完整玩家路径？
每个资源是否有来源、存放和去向？
用户是否已经明确指出哪些资源可以背在角色身后，哪些资源不能？
每个交互对象是否有触发规则和结果？
每个升级是否有消耗、触发和完成效果？
每个阶段是否有开始条件和完成条件？
每个参与 first playable 的对象是否都有 binding、zone 或等价查询方式？
缺少 binding 时是否已经向用户确认补 binding 或 fallback？
指定资产是否都有 gameplay 角色，或明确为 Decoration？
地编区域是否都有玩法意义，或明确为装饰区域？
结束条件、CTA、Endcard、跳转是否清楚？
```

## 6. Missing Flow Patch

如果美术发现 `gameplay.md` 漏掉流程，不应只反馈“流程不完整”。应按下面格式补充，agent 根据 patch 局部修正文档。

```md
## Missing Flow Patch

### 缺失流程名称

### 来源

- xlsx / sheet / cell：
- 参考 HTML / 时间点：
- 截图 / 分镜：
- 场景节点：

### 当前 gameplay.md 的问题

- 完全没写 / 写错 / 顺序错 / 触发条件缺失 / 结束条件缺失

### 正确流程

按玩家视角描述正确流程。

### 涉及资产 / 场景对象

- 资源：
- NPC / 顾客 / 机器：
- 交互区域：
- 解锁区域：

### 是否阻塞

- 阻塞 / 非阻塞
- 原因：
```

## 7. Agent 边界

`gameplay` agent 可以整理、追问、修正 `gameplay.md`，但不能：

```text
写 gameplay 代码
生成实现计划
修改 scene.json、binding 或源代码
把 AI 推测写成确定需求
自行推断哪些资源可以背在角色身后
用默认规则覆盖美术或需求方明确要求
```

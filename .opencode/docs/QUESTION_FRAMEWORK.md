# 0. Gameplay Question Framework

权威问题框架维护在 `wiki/sources/docs/guides/gameplay/QUESTION_FRAMEWORK.md`。本文件只是 `.opencode` 兼容副本。

## 1. 文档目标

本文档定义 `gameplay knowledge base` 的最终问题清单。

它服务于后续在 `forge-play` 中新增 `gameplay` 模式：该模式的目标不是直接实现玩法，而是帮助用户把玩法想法整理成一份可供美术、程序和 AI 后续执行的 `gameplay.md` / `gameplay knowledge base`。

本文档只规定问题框架和回答规则，不规定最终 UI、不规定平台实现方式，也不绑定某个具体 PA 项目。

## 2. 通用核心循环

所有初版 PA gameplay 都应优先回答同一条核心循环：

```text
玩家进入场景
-> 找到资源来源
-> 通过操作获得资源
-> 将资源带到目标区域
-> 资源沿转换链条变成钱
-> 消耗资源进行升级
-> 升级提升效率、容量、范围、产钱效率或自动化程度
-> 玩家进入更高效率的新一轮循环
-> 达成所有阶段目标
```

不同项目可以在表现上不同：

1. 玩家主体可以是角色、载具、机器或其他可交互系统
2. 资源是广义资源，可以是可收集物、中间产物、交易获得的钱、订单进度或其他可消耗数值
3. 交付方式可以是售卖、加工、配送、装车、提交订单或完成路线
4. 升级是广义升级，可以是工具强化、容量提升、区域解锁、新能力、新机器、NPC、载具、防御或自动化

但 `gameplay knowledge base` 必须把这条循环写清楚。

## 3. 阶段与升级规则

游戏可以根据阶段进行拆解，可以只有一个阶段，也可以有多个阶段。

阶段与升级遵循下面的默认规则：

1. 阶段由一组升级定义，而不是由任意剧情段落定义。
2. 每次升级都应带来明确变化，例如效率、容量、范围、产钱效率、区域、新能力、新机器、NPC、载具、防御或自动化变化。
3. 阶段内的升级通常是自由并行可选的，玩家可以自行决定完成顺序。
4. 阶段完成条件通常是该阶段内所有必需升级全部完成。
5. 阶段完成时也可能带来阶段级变化，例如新资源链条、新区域、新机器、新能力、新路线、新敌人或更高效率循环。
6. 如果存在二选一升级，未选择的升级通常也应在后续游玩流程中重新开放，而不是永久不可选。
7. 纯可选升级如果不影响阶段推进，不应计入阶段完成条件。

## 4. 问题回答规则

问题清单使用下面的标记规则：

1. `问题列表` 中默认问题为必答问题。
2. `可选问题` 分类中的问题为可选回答问题。
3. 必答问题必须在最终 `gameplay.md` 中得到明确答案。
4. 可选问题可以在信息不足或项目不适用时跳过。
5. 如果必答问题暂时无法确定，`gameplay` 模式必须追问用户；在草稿阶段可以临时标记为 `待确认`，但最终版 `gameplay.md` 不应留下未回答的必答问题。
6. 如果问题可由平台默认规则、地编结果或已知上下文回答，可以由 AI 直接补全，但仍应在文档中留下明确结论。

## 5. 问题列表

### 5.1 游戏定位

这类问题用于确定项目的玩法边界。游戏结束条件和失败条件统一放到 `核心循环` 中回答。

1. 这个游戏的一句话玩法是什么？

### 5.2 3C

这类问题用于定义玩家实际的交互规则。

1. Character：玩家控制的主体是什么？如果没有需要特别说明。
2. Control：玩家如何进行操作？
3. Camera：镜头如何跟随或展示主体？

### 5.3 资源链路

这类问题用于先定义 gameplay 中有哪些资源，再定义资源如何存放、转换和产钱。资源主链必须从可收集资源开始，经过可选的中间资源，最终导向钱；同时允许部分资源作为旁支，不进入钱的转换链条，而是在升级列表中被消耗。

1. 游戏中有哪些资源？分别属于可收集资源、中间资源、钱或其他旁支资源？
2. 每种可收集资源从哪里来？玩家如何获得它？
3. 每种资源会存放在哪里？是否有容量限制？
4. 资源转换链条是什么？
   - 形式示例：可收集资源 -> 中间资源 -> 钱
   - 链条可以有多条，也可以没有中间资源，但资源主链必须从可收集资源开始并最终导向钱
   - 旁支资源可以直接用于升级，不一定先转换成钱，但具体消耗关系应放到核心循环的升级列表中回答
5. 链条中的每一步资源转换在哪里发生？由玩家、机器、NPC、车辆、订单、顾客、传送带还是其他自动化设施完成？
6. 钱在哪一步产生？
7. 哪些资源数值需要在 HUD 中显示？
8. 每一步资源获取、存放、转换和产钱的数量规则是什么？
9. 每一步资源获取、存放、转换和产钱的触发规则是什么？

### 5.4 核心循环

这类问题用于判断玩法闭环是否成立，并统一回答阶段和升级问题。升级是广义升级，包含区域解锁、新能力、新机器、效率提升等；升级一定会消耗资源。所有阶段都达成时，即认为游戏结束。

1. 游戏分为几个阶段？每个阶段的开始条件和完成条件是什么？
2. 每个阶段使用哪条资源链路？阶段切换后资源链路是否发生变化？
3. 每个阶段内有哪些升级？
4. 每个升级分别消耗什么资源？
5. 每个升级的触发位置或触发方式是什么？
6. 每个升级完成后分别带来什么变化？
7. 每个升级是否为阶段完成所必需？
8. 阶段内升级是否自由并行可选？
9. 完成一个阶段后，会带来什么阶段级变化？
10. 如果存在二选一升级，未选择的升级是否会在后续流程中重新开放？
11. 游戏是否有失败条件？
12. 初版 gameplay 的最小完整游玩路径是什么？玩家完成到哪一步算初版闭环完成？

### 5.5 Binding 需求

这类问题用于确保美术、地编、文档 AI 和开发 AI 对 gameplay binding 的责任一致。只要对象参与 first playable 主路径，就必须回答。

1. 哪些 gameplay 对象需要明确 binding、zone、spawn point、path point 或 runtime node reference？
2. 每个需要 binding 的对象期望的 logicType 或等价玩法类型是什么？
3. 每个 binding 需要哪些字段才能实现，例如 resourceType、acceptsResourceTypes、producesResourceTypes、capacity、cost、dependsOn、unlocks、spawnRootId、pathPointIds？
4. 哪些对象目前由美术或地编负责补 binding？
5. 哪些对象允许开发临时使用 fallback？fallback 需要用户明确确认，不能由 AI 静默决定。
6. 缺少哪些 binding 会阻塞 first playable 主路径？
7. 如果 binding 缺失，应该问用户什么具体问题？

### 5.6 可选问题

这类问题不影响初版 gameplay 文档成立，可以在信息不足或项目不适用时跳过。

1. 这个游戏更接近哪种玩法形态？

## 6. 对 gameplay 模式的约束

后续 `gameplay` 模式生成文档时，应遵循以下约束：

1. 不直接写 gameplay 代码
2. 不直接修改 gameplay 文档以外的项目实现文件
3. 不替用户跳过关键玩法问题
4. 不把所有 PA 强行写成同一种题材模板
5. 必须围绕核心循环组织内容
6. 必须显式写出假设项；草稿阶段允许保留待确认项，最终版不得留下未回答的必答问题
7. 必须输出可被美术、程序和后续 AI 模式共同理解的 `gameplay.md`
